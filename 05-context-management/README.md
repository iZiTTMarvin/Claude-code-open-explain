# 05 — 上下文管理与压缩

> Claude Code 不是只有一个聊天窗口，它还要长期管理 prompt、历史消息、工具结果和压缩摘要

## 为什么这一章必须看

很多人第一次接触 AI Agent 时，会有一个天真的想法：**”模型不是有很大的上下文窗口吗？直接把所有历史都发过去不就行了？”**

这个想法在理论上没错，但在实践中会遇到三个致命问题：

**问题 1：窗口再大也有上限**

即使是最新的大模型，上下文窗口也就是几十万到上百万 token。听起来很多，但在真实编程场景下，这个窗口会被迅速填满：

- 读一个中等规模的文件：几千 token
- 执行一次 Shell 命令的输出：几百到几千 token
- 一轮完整的工具调用（请求+结果）：几千 token
- 多轮对话累积：几万 token

如果你和 Claude Code 协作一个小时，轻松就能产生几十万 token 的历史。窗口再大，也会被填满。

**问题 2：成本和延迟会失控**

每次调用模型，你都要为”输入的 token 数量”付费。如果每次都把所有历史发过去：

- 第 1 轮：1000 token → 便宜
- 第 10 轮：10000 token → 开始贵了
- 第 50 轮：50000 token → 非常贵
- 第 100 轮：100000 token → 难以承受

而且 token 越多，模型处理越慢。用户会明显感觉到”越聊越卡”。

**问题 3：信息密度会下降**

更隐蔽的问题是：历史越长，模型越难抓住重点。

想象一下，你要在一本 1000 页的书里找一句话，和在一本 10 页的书里找一句话，哪个更容易？模型也是一样——上下文越长，它越容易被无关信息干扰。

**所以，Claude Code 真正要解决的问题不是”有没有上下文”，而是：**

> 在上下文持续增长的情况下，怎样让模型始终看到最有价值、最可继续工作的那部分信息，同时控制成本和延迟。

这就是为什么上下文管理不是”锦上添花”，而是”这类 Agent 系统的基础设施”。

## 本章关键源码入口

| 文件 | 作用 |
|------|------|
| `src/utils/queryContext.ts` | 获取本轮请求需要的上下文三件套 |
| `src/context.ts` | 生成用户上下文与系统上下文 |
| `src/services/compact/autoCompact.ts` | 自动压缩阈值与触发逻辑 |
| `src/services/compact/compact.ts` | 完整压缩流程 |
| `src/services/compact/microCompact.ts` | 更细粒度的微压缩能力 |

## 先分清：Claude Code 里的上下文到底有哪些

### 1. Prompt 类上下文

这部分决定模型“该怎么做”。

- `systemPrompt`
- 输出风格
- 工具使用规则
- 风险操作约束

### 2. 环境类上下文

这部分决定模型“现在身处什么环境”。

- 操作系统
- 当前目录
- Git 状态
- 可能的附加工作目录

### 3. 项目类上下文

这部分决定模型“当前项目有什么额外规则”。

- `CLAUDE.md`
- 目录级规则
- 记忆类内容

### 4. 历史交互类上下文

这部分决定模型“前面已经做过什么”。

- 过去的用户消息
- 过去的 assistant 响应
- 之前的工具调用结果
- 之前的压缩摘要

只有把这几类都分开，后面的压缩策略才说得通。

## `fetchSystemPromptParts()` 很值得注意

在 `src/utils/queryContext.ts` 里，Claude Code 会一次性取出：

- `defaultSystemPrompt`
- `userContext`
- `systemContext`

这件事本身就说明了一个设计态度：

> 上下文不是一锅粥，而是分通道管理。

这样做的好处是：

- 不同信息可以用不同方式注入
- 更利于做缓存
- 更利于后续做压缩和替换

### 这里再补一个更具体的事实

它不是“按顺序取 3 份东西”，而是并发取：

- `getSystemPrompt(...)`
- `getUserContext()`
- `getSystemContext()`

而且如果调用方已经给了 `customSystemPrompt`，这条路径会直接跳过默认 prompt builder 和 `getSystemContext()` 的默认装配逻辑。

这说明 Claude Code 对上下文装配的理解不是“每轮全算一遍”，而是：

> 能并发的并发，能绕开的绕开，尽量把真正需要的上下文算到位。

## 为什么长对话一定要做压缩

只要 Claude Code 真的在帮你做事，上下文很快就会膨胀：

- 读文件会带来大量文本
- 工具结果可能很长
- 多轮推理会不断累积历史
- 每次新请求都要带上一部分旧信息

如果没有压缩策略，系统迟早会遇到两个问题：

1. 模型窗口被塞满
2. 成本和延迟迅速上涨

所以压缩不是锦上添花，而是这类 Agent 系统的基础设施。

## 自动压缩不是随便触发的

在 `src/services/compact/autoCompact.ts` 里，你能看到一套比较细的阈值逻辑。

源码里并不是简单写成：

```text
超了就压
```

而是会先考虑：

- 当前模型的上下文窗口有多大
- 要给输出预留多少空间
- 自动压缩缓冲区是多少
- 是否允许自动压缩
- 某些特殊模式下是否要抑制自动压缩

这说明 Claude Code 对压缩的理解不是“最后兜底”，而是“运行时持续调度的一部分”。

### 阈值计算比“超了就压”更具体

从 `autoCompact.ts` 的实现看，至少有三层数值关系：

1. 先从模型上下文窗口里预留一段给 compact summary 输出
2. 再减掉 `AUTOCOMPACT_BUFFER_TOKENS`
3. 还要和 blocking limit、warning/error buffer 区分开

所以更准确的说法是：

```text
原始窗口
  -> 预留 summary 输出空间
  -> 得到 effective context window
  -> 再减去 proactive buffer
  -> 得到 auto compact threshold
```

## 完整压缩和微压缩有什么区别

### 1. 完整压缩

完整压缩更像一次较大的整理动作：

- 摘要旧消息
- 生成压缩边界
- 用更短的摘要替代更长的历史内容
- 让后续会话还能继续理解之前发生过什么

这类压缩解决的是“历史太厚”的问题。

### 2. 微压缩

微压缩更像局部削薄：

- 优先处理超大的工具结果
- 尽量保住主结构
- 不必每次都做一次完整摘要

这类压缩解决的是“某些局部内容特别肥”的问题。

所以完整压缩和微压缩不是谁替代谁，而是两种粒度不同的上下文治理方式。

### 还有一个常被忽略的点：并不是所有 query source 都允许 auto compact

`shouldAutoCompact()` 里至少会显式跳过：

- `session_memory`
- `compact`

某些特性打开时，还会继续跳过像 `marble_origami` 这样的来源。

这说明 Claude Code 不是“只要长就压”，而是会先判断：

> 这一轮压缩会不会和当前上下文管理机制自己打架。

## 为什么 Claude Code 不把压缩理解成”删历史”

这是一个很重要的产品心态差异。

**粗糙的实现会这样做：**

```text
上下文快满了
  ↓
删掉最早的 50% 消息
  ↓
继续
```

这样做确实能腾出空间，但会带来严重问题：

**问题 1：任务连续性断裂**

想象你正在和 Claude Code 一起重构一个复杂模块。你们已经讨论了架构、确定了方案、修改了几个文件。突然，因为上下文满了，系统把前面的讨论全删了。

下一轮，模型会完全忘记：
- 你们为什么要这样重构
- 已经改了哪些文件
- 还有哪些文件需要改

它会像失忆一样，要么重复之前的工作，要么做出不一致的决策。

**问题 2：关键信息丢失**

不是所有历史消息的价值都一样。有些消息包含关键决策：

- “我们决定用 Redis 而不是 Memcached”
- “这个 bug 的根因是并发竞争”
- “用户要求必须兼容 IE11”

如果简单按时间删除，这些关键信息可能被删掉，而一些无关紧要的闲聊反而被保留。

**问题 3：用户体验崩溃**

从用户角度看，这就像是：

```text
你：我们继续刚才的重构吧
Claude Code：什么重构？我不记得了
你：？？？
```

这种体验是灾难性的。用户会觉得系统”不靠谱”，不敢把复杂任务交给它。

### Claude Code 的解决方案：压缩而不是删除

Claude Code 的做法更像是”历史重组”：

**1. 生成摘要**

不是删掉旧消息，而是让模型生成一个简洁的摘要：

```text
原始历史（10000 token）：
  用户：我想重构 auth 模块
  助手：好的，我先分析一下现有代码...
  [读取文件]
  助手：我发现了三个问题...
  用户：第二个问题最严重，我们先解决它
  助手：好的，我建议...
  [多轮讨论]

压缩后（1000 token）：
  用户和助手讨论了 auth 模块重构。
  主要问题是并发安全性。
  决定采用 Redis 分布式锁方案。
  已修改 auth.ts 和 lock.ts。
  下一步需要修改 session.ts。
```

**2. 保留边界信息**

系统会在压缩点插入一个标记，告诉模型”这里之前的内容已经被压缩了”。这样模型知道：
- 有些信息不在当前上下文里
- 但可以通过摘要了解大致情况
- 如果需要更多细节，可以请求用户补充

**3. 维持任务连续性**

压缩后，模型仍然能够：
- 理解当前在做什么
- 知道之前做过什么
- 继续推进任务

只是细节变少了，但主线还在。

### 从主循环角度看：压缩是”无缝续跑”

这是很多人容易忽略的一个细节。从 `query.ts` 的实现来看，auto compact 成功后，系统不是”结束当前轮次，让用户重新开始”，而是：

```text
1. 触发压缩
   ↓
2. 生成压缩摘要
   ↓
3. 重建 postCompactMessages
   ↓
4. 用新消息替换当前 query 的 messages
   ↓
5. 继续当前这轮 query
```

**用户感知到的是”无缝续跑”，而不是”会话被强行截断重开”。**

这就像是：你在高速公路上开车，油快用完了。系统不是让你停车、下车、走回加油站，而是在你开车的同时，悄悄把油箱换成了一个新的、更高效的油箱。你甚至不需要停车。

### 这种设计的代价

当然，这种”压缩而不是删除”的方案也有代价：

**1. 需要额外的模型调用**

生成摘要本身需要调用模型，这会增加一些成本和延迟。

**2. 摘要可能不完美**

模型生成的摘要可能会遗漏一些细节，或者理解有偏差。

**3. 系统复杂度增加**

压缩机制需要维护边界信息、管理元数据、处理各种边界情况，代码复杂度会上升。

**但 Claude Code 认为这些代价是值得的。**

因为如果不这样做，系统就只能在”长对话”和”任务连续性”之间二选一。而压缩机制让两者可以兼得——虽然不完美，但足够实用。

## `CLAUDE.md` 为什么也是上下文管理的一部分

很多人把 `CLAUDE.md` 看成静态配置，但从运行时角度看，它其实是上下文注入的一部分。

这意味着：

- 它会参与模型理解
- 它会影响缓存
- 它会影响压缩后的继续运行

也正因为如此，Claude Code 才不愿意把它粗暴塞进稳定 prompt 前缀，而是单独作为更适合变化的上下文来源处理。

## 为什么压缩要和会话状态绑定

压缩不是一个完全独立的小工具，它会直接影响：

- 当前消息历史
- 后续 token 预算
- 工具结果引用关系
- 某些发现过的技能或工具集合

这也是为什么 `compact.ts` 不只是“调一个摘要模型然后替换文本”，而是要连带维护很多会话元数据。

## 新手可以怎么理解“无限上下文”

Claude Code 在系统提示里会告诉模型，系统有自动摘要机制，所以对话可以持续下去。

这句话的正确理解不是：

> 模型真的拥有无限窗口。

而是：

> 运行时会不断把旧信息压缩成更紧凑的形式，让协作尽可能延续。

这是一种工程意义上的“近似无限”，不是物理意义上的无限。

## 哪些情况下压缩反而会变难

这也是读源码时很值得注意的一点。

上下文越复杂，压缩越不只是“做摘要”：

- 工具结果可能包含结构化信息
- 附件可能要在压缩后重新注入
- 子 Agent 可能有自己独立的上下文
- 某些模式下压缩会和缓存、会话恢复相互影响

这说明上下文管理从来不是一个孤立模块，而是和 Prompt、Tool、Session、Cache 都强耦合。

### compact 和 prompt cache 也会互相牵制

这点在 `streamCompactSummary()` 一类路径里很明显：

- 某些路径优先保证 prompt-cache key 稳定
- 某些 fallback 路径才显式限制 summary 输出上限

所以 compact 不是纯粹的“上下文摘要问题”，它还会反过来影响缓存命中策略。

## 新手常见误区

### 误区 1：上下文管理只和模型窗口大小有关

不对。它还和工具结果、缓存、会话恢复、UI 状态都有关系。

### 误区 2：压缩就是摘要旧消息

不对。Claude Code 还会保留边界信息、维护元数据、处理局部瘦身。

### 误区 3：`CLAUDE.md` 只是静态配置文件

不对。从运行时角度看，它也是被注入、被管理的一部分上下文。

### 误区 4：压缩触发后，这轮对话就结束了

不对。很多时候 compact 是当前 query 内部的一次状态跃迁，压完以后会直接继续跑。

## 本章小结

Claude Code 的上下文管理做了几件很成熟的事：

- 把不同来源的上下文分通道管理
- 在窗口接近上限前主动治理
- 同时提供完整压缩与细粒度微压缩
- 不把压缩等同于“删除历史”，而是尽量保住任务连续性
- 阈值计算会先预留 summary 输出空间，再决定 proactive compact 时机
- compact 会和 query source、prompt cache、恢复路径一起共同决定是否继续
