# 04 — 权限安全模型

> Claude Code 不是“模型想调什么就调什么”，每次工具调用前都要先过权限系统

## 为什么这一章很重要

想象一下这个场景：

你让 Claude Code 帮你”清理项目里的临时文件”。模型理解了你的需求，决定执行 `rm -rf tmp/`。但如果它不小心把路径写成了 `rm -rf /`，会发生什么？

**你的整个系统会被删除。**

这不是危言耸听，而是真实存在的风险。模型再聪明，也会犯错——它可能误解需求、拼错路径、选错工具。如果没有权限系统，这些错误会直接变成灾难。

**这就是为什么权限系统不是”可有可无的保护层”，而是”让 Claude Code 能够落地的必要前提”。**

没有权限系统，Claude Code 就只是一个”把高权限本地能力直接交给模型的危险程序”。有了权限系统，它才能成为”既能做事、又不会失控的编程助手”。

真正让它能够落地的关键，不只是工具多，而是：

> 工具在执行前会被严格判定”这次能不能做、该不该问、有没有超范围”。

而且这个判定不是简单的”允许/拒绝”，而是一套多层防线：先看模式、再看规则、再分析参数、最后才决定。

## 本章关键源码入口

| 文件 | 作用 |
|------|------|
| `src/utils/permissions/PermissionMode.ts` | 定义权限模式 |
| `src/utils/permissions/permissionSetup.ts` | 初始化和切换权限上下文 |
| `src/utils/permissions/permissions.ts` | 规则匹配与权限判定 |
| `src/utils/permissions/pathValidation.ts` | 文件路径安全验证 |
| `src/utils/permissions/denialTracking.ts` | 拒绝记录与追踪 |
| `src/utils/permissions/yoloClassifier.ts` | 自动模式下的分类器逻辑 |

## 新手先记住：权限不是一个简单的 allow / deny

Claude Code 的权限系统更像多层判定流水线：

```text
模型请求调用工具
  ↓
先看当前权限模式
  ↓
再看工具本身的权限语义
  ↓
再看输入参数是否敏感
  ↓
再看路径是否合法
  ↓
再看是否命中允许/拒绝规则
  ↓
最后决定：
  - 自动允许
  - 提示用户确认
  - 直接拒绝
```

所以你不能把它理解成“一个 if 判断”。

## 实际有哪些权限模式

从源码里的 `PermissionMode.ts` 看，至少可以看到这些模式：

- `default`
- `plan`
- `acceptEdits`
- `bypassPermissions`
- `dontAsk`

在某些带条件开关的场景下，还会出现：

- `auto`

### 这里还要再补一个关键事实

权限模式不是等到模型真的调工具时才临时决定的。  
从 `main.tsx` 和 `initialPermissionModeFromCLI(...)` 的调用链看，它是在 **CLI 启动阶段** 就先被归一化和裁决的。

大体顺序是：

```text
dangerouslySkipPermissions
  ->
CLI 传入的 permission mode
  ->
settings 里的默认 permission mode
  ->
归一化为最终 mode
```

这意味着：

- 未知 mode 不会一路传到底层
- 被 gate 禁用的 mode 会提前收敛
- 后续工具执行看到的是已经裁定好的 mode

### 这些模式可以先这样理解

#### `default`

默认模式。  
读操作通常更容易自动放行，写操作、命令执行、外部动作更容易触发确认。

#### `plan`

计划模式。  
它的核心目标不是高效率，而是先分析、先规划、减少直接改动。

#### `acceptEdits`

从名字就能看出，它比 `default` 更激进，重点是减少编辑类动作上的频繁确认。

#### `bypassPermissions`

几乎就是“绕过常规权限确认”。  
这类模式威力很大，因此也最需要额外谨慎。

#### `dontAsk`

这类模式体现了 Claude Code 内部权限系统不仅服务交互式 CLI，也要服务更自动化、更程序化的调用场景。

#### `auto`

这不是简单的“全部自动允许”，而是和专门的自动模式判定逻辑绑在一起。  
换句话说，自动模式也不是无脑放权。

## 权限模式为什么单独做成系统

因为 Claude Code 需要同时支持多种使用方式：

- 交互式终端
- 更自动化的执行流
- 计划模式
- SDK / 非交互场景
- 多 Agent 协作

如果没有一层统一的权限模式抽象，这些场景会把工具系统撕得很碎。

## 路径校验为什么是权限系统的核心组成部分

很多人第一次看权限系统时，会把注意力放在”工具名”上：这个工具允许不允许？但这样理解会错过最关键的风险点。

**真正危险的往往不是”工具本身”，而是”工具操作的路径”。**

让我们看几个例子，同样是”写文件”这个工具：

**场景 1：写项目代码**
```
FileWriteTool → src/app.ts
```
这是正常的编程操作，风险很低。

**场景 2：写 Git 钩子**
```
FileWriteTool → .git/hooks/post-commit
```
这就危险了。Git 钩子会在每次提交后自动执行，如果写入恶意代码，后果严重。

**场景 3：写系统敏感目录**
```
FileWriteTool → /etc/passwd
```
这是灾难性的。直接修改系统文件，可能导致整个系统无法使用。

**看到了吗？同样是 FileWriteTool，风险完全不同。**

所以 Claude Code 的权限系统不能只看”工具名”，还必须深入到”参数层”——特别是路径参数。

### 路径校验到底在检查什么？

从源码实现来看，路径校验至少关注这几个维度：

**1. 是否越过允许目录**
Claude Code 通常限定在当前工作目录内操作。如果路径指向工作目录之外，就需要特别警惕。

**2. 是否尝试路径逃逸**
像 `../../../etc/passwd` 这样的路径，看起来是相对路径，实际上可能逃逸到系统敏感目录。

**3. 是否命中内部受保护文件**
`.git/`、`.env`、`node_modules/` 这类目录，通常不应该被随意修改。

**4. 是否在自动编辑场景下触碰危险位置**
在自动模式下，某些路径的风险会被放大——因为没有人工确认环节，一旦出错就直接执行了。

### 为什么路径边界比 allow 规则还优先？

这是很多人容易误解的地方。你可能会想：”我已经允许了 FileWriteTool，为什么还要检查路径？”

**因为”允许工具”不等于”允许工具做任何事”。**

从源码的判定顺序来看：

```text
1. 先判定这条路径本身安不安全
   ↓
2. 如果路径本身就危险，直接拒绝
   ↓
3. 如果路径相对安全，再看工作目录、allow 规则、sandbox allowlist
   ↓
4. 最后才决定是否允许
```

换句话说：**路径边界是”硬约束”，allow 规则是”软策略”。**

即使你在配置里写了”允许所有文件写入”，系统也不会让你写 `/etc/passwd`。这不是 bug，而是设计——某些边界不应该被配置覆盖。

### 一个真实的设计取舍

你可能会问：”这样不是限制太死了吗？万一我真的需要操作系统文件呢？”

**这就是 Claude Code 的设计哲学：宁可限制过严，也不要留下安全漏洞。**

如果你真的需要操作系统敏感文件，正确的做法不是”绕过路径检查”，而是：
1. 先用普通权限完成大部分工作
2. 对于必须提权的操作，显式切换到更高权限模式
3. 操作完成后，立刻切回普通模式

这种”最小权限原则”，是所有安全系统的基础。Claude Code 只是把它认真落实了而已。

## 为什么 Bash 命令要走更重的权限链路

因为 Shell 命令是最强、也最危险的工具之一。

同样是 `BashTool`，风险差异极大：

- `pwd`
- `git status`
- `cat README.md`

和：

- `rm -rf`
- `git push --force`
- 重定向写文件
- 管道执行不可信脚本

完全不是一个风险等级。

所以 Claude Code 对命令的处理不会停在“这是 BashTool，所以都一样”，而是继续分析：

- 这是读命令还是写命令
- 是否可能造成破坏
- 是否访问了敏感路径
- 当前模式是否允许这类动作

### 自动模式还会主动剥离危险规则

这里再补一个容易被忽略的事实：  
Claude Code 不只是“执行前再判定 Bash 命令”，它还会在 auto mode 入口就把某些危险 allow 规则剥掉。

例如：

- 空规则
- `*`
- 某些解释器/执行器前缀
- 某些 PowerShell 启动与表达式执行形状

这说明它不是只在运行时临场防守，而是：

> 在进入自动化模式前，先把最容易导致越权的规则形状清理掉。

## 规则系统不是摆设

除了模式本身，Claude Code 还有规则层。

规则层的作用是让权限不只是依赖“当前模式”，还可以根据更细粒度条件判定，例如：

- 某类工具默认允许
- 某类路径总是拒绝
- 某类命令必须询问

这让权限系统从“模式切换”升级成“策略系统”。

### 但这里还有一个很关键的顺序问题

很多人会以为“命中 allow 规则就直接放行”。  
源码里的真实顺序更保守：

- 先看 deny
- 再看 ask
- 再跑工具自己的权限检查
- 如果命中 `safetyCheck`，优先直接拦住
- 之后才轮到 bypass、whole-tool allow 这些更宽松机制

也就是说：

> 规则、模式、工具内部检查，不是并列关系，而是按优先级串成一条流水线。

## 自动模式为什么还需要分类器

很多人会误以为“自动模式 = 全放开”，源码不是这么设计的。

在自动模式相关逻辑里，你能看到专门的分类器模块，例如 `yoloClassifier.ts`。这说明自动模式并不是放弃判断，而是：

> 把“是否应该自动放行”本身也做成一个需要分析的过程。

这非常关键，因为真正可用的自动化，不是胆子大，而是知道什么时候该停。

而且从危险规则剥离那部分实现可以看出，Claude Code 的思路不是“先全放开，分类器再兜底”，而是：

- 先把明显危险的 allow 形状剥掉
- 再让自动模式和分类器继续工作

这说明自动模式的设计重点是**减少提前越权**，而不是单纯追求省确认。

## 权限拒绝为什么还要被记录下来

`QueryEngine.ts` 里会持续记录 `permissionDenials`。

这件事的意义至少有 3 层：

1. UI 或 SDK 可以知道到底哪一步被拒绝了
2. 后续系统可以针对拒绝行为做反馈和分析
3. 模型在下一步可能会调整策略，而不是死磕同一个被拒动作

这说明 Claude Code 的权限系统不是“挡住就完了”，而是整个对话状态的一部分。

## 计划模式和普通模式为什么要区分

这其实体现了 Claude Code 对“工作阶段”的理解。

有些时候，用户想要的是：

- 先分析
- 先列计划
- 先解释影响

而不是立刻改代码。

所以 `plan` 这类模式不是性能优化，而是交互语义优化。  
它让系统知道，现在更像“顾问”而不是“直接执行者”。

### 这和 `bypassPermissions` 也不是一个方向

- `plan` 更像先分析、先规划
- `bypassPermissions` 更像减少常规确认

把它们都叫“更自由”会误导读者。  
它们服务的是不同工作阶段，不是同一把油门大小不同。

## 新手应该如何看待这套权限系统

可以把它理解成 3 层防线：

### 第一层：模式

决定大方向，是偏保守、偏计划，还是偏自动。

### 第二层：规则

决定某些工具、路径、命令的细粒度策略。

### 第三层：参数分析

决定这一次具体输入到底是不是危险。

这 3 层叠在一起，Claude Code 才真正具备“既能做事，又不至于失控”的基础。

如果按更贴近源码的说法，也可以把它改写成：

1. 启动期先裁定 permission mode
2. 规则层再决定 deny / ask / allow 的形状
3. 工具级权限检查处理具体输入
4. 路径与 safetyCheck 进一步拦危险动作

## 新手常见误区

### 误区 1：权限只是 UI 上那个确认弹窗

不对。弹窗只是最后一层交互表现，前面已经做了很多分析。

### 误区 2：给了某个工具权限，就等于所有同类操作都安全

不对。路径、参数、模式、规则都可能让同一个工具在不同场景下得到不同结论。

### 误区 3：自动模式就是彻底放权

不对。自动模式真正难的地方恰恰是“自动但仍然有判断”。

## 本章小结

Claude Code 的权限系统不是给模型套一个简陋开关，而是：

- 用模式定义大方向
- 用规则补充策略
- 用参数分析处理具体风险
- 用记录机制把拒绝纳入会话状态
- 启动期先裁定 mode，而不是把 mode 推迟到执行瞬间
- safetyCheck 和路径边界优先级高于很多看起来更宽松的 allow 机制

这也是它能在真实终端环境里落地的关键前提。
